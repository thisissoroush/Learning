# Chapter 2: Async Basics

> *"Once you start using async, it's best to allow it to grow through your code."*

---

## Core Concept

`async`/`await` transforms naturally sequential-looking code into non-blocking code. The compiler generates a state machine behind the scenes — your code reads top-to-bottom, but threads are never wasted waiting for I/O. This chapter is a collection of the most common async patterns you'll need day-to-day.

```
How async/await works conceptually:
─────────────────────────────────
  async Task DoWorkAsync()            State machine generated by compiler:
  {                                   ┌─────────────────────────────────┐
    // Part 1 (runs synchronously)    │ State 0: run Part 1             │
    await SomeOperationAsync();   →   │          await → pause method   │
    // Part 2 (resumes on context)    │ State 1: resume when done       │
  }                                   │          run Part 2             │
                                      └─────────────────────────────────┘
```

## Recipe 2.1 — Pausing for a Period of Time

**Problem:** You need to asynchronously wait for some duration (retry delays, timeouts, unit test faking).

**Solution:** `Task.Delay()`

```csharp
// Simple delay
await Task.Delay(TimeSpan.FromSeconds(1));

// Exponential backoff — doubles the delay on each retry
async Task<string> DownloadWithRetries(HttpClient client, string url)
{
    TimeSpan nextDelay = TimeSpan.FromSeconds(1);
    for (int i = 0; i != 3; ++i)
    {
        try { return await client.GetStringAsync(url); }
        catch { }

        await Task.Delay(nextDelay);
        nextDelay = nextDelay + nextDelay;   // 1s → 2s → 4s
    }
    return await client.GetStringAsync(url); // final attempt, let it throw
}

// Soft timeout — returns null if no response in 3 seconds
async Task<string> DownloadWithTimeout(HttpClient client, string url)
{
    using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(3));
    Task<string> download = client.GetStringAsync(url);
    Task timeout = Task.Delay(Timeout.InfiniteTimeSpan, cts.Token);

    Task completed = await Task.WhenAny(download, timeout);
    if (completed == timeout)
        return null;   // timed out

    return await download;
}
```

> 💡 For production retry logic, prefer the **Polly** NuGet library — it handles policies, circuit breakers, and more.

> ⚠️ The soft timeout approach doesn't *cancel* the operation — the download continues in the background. Prefer `CancellationToken` passed directly to the operation when possible.

## Recipe 2.2 — Returning Completed Tasks

**Problem:** You need to implement an async interface signature with synchronous code (e.g., for testing stubs).

**Solution:** `Task.FromResult`, `Task.CompletedTask`, `Task.FromException`, `Task.FromCanceled`

```csharp
// Successful synchronous implementation
public Task<int> GetValueAsync()
    => Task.FromResult(42);

// Void-returning synchronous implementation
public Task DoSomethingAsync()
    => Task.CompletedTask;

// Returning a pre-faulted task
public Task<T> NotImplementedAsync<T>()
    => Task.FromException<T>(new NotImplementedException());

// Returning a pre-canceled task
public Task<int> GetValueAsync(CancellationToken ct)
{
    if (ct.IsCancellationRequested)
        return Task.FromCanceled<int>(ct);
    return Task.FromResult(42);
}

// Synchronous implementation that may fail
public Task DoSomethingAsync()
{
    try
    {
        DoSomethingSynchronously();
        return Task.CompletedTask;
    }
    catch (Exception ex)
    {
        return Task.FromException(ex);  // wrap in a faulted task
    }
}
```

> 💡 **Cache frequently-used completed tasks** to avoid allocations:
> ```csharp
> private static readonly Task<int> ZeroTask = Task.FromResult(0);
> ```

> ⚠️ **Never block inside an async method signature implementation.** Don't do `Thread.Sleep()` and return a completed task — that blocks the calling thread while pretending to be async.

## Recipe 2.3 — Reporting Progress

**Problem:** You have a long-running async operation and need to report progress to the UI.

**Solution:** `IProgress<T>` and `Progress<T>`

```csharp
// Method that reports progress
async Task ProcessFilesAsync(IProgress<double> progress = null)
{
    var files = GetFiles();
    for (int i = 0; i < files.Count; i++)
    {
        await ProcessFileAsync(files[i]);
        // Report % complete; null-check: caller may not want progress
        progress?.Report((double)i / files.Count * 100);
    }
}

// Calling code — Progress<T> captures the current context (UI thread)
async Task ButtonClick_Handler()
{
    var progress = new Progress<double>();
    progress.ProgressChanged += (_, pct) =>
        ProgressBar.Value = pct;  // safe to update UI here

    await ProcessFilesAsync(progress);
}
```

```
Context capture by Progress<T>:
  UI Thread creates Progress<T> ─────────────────────────┐
                                                         │ captures SynchronizationContext
  Background thread calls Report() ────────────────────→ │
                                                         ▼
                                          ProgressChanged fires on UI thread ✓
```

> 💡 Make `T` an **immutable type or value type** — `Report()` is usually asynchronous, so `MyMethodAsync` may continue before the progress is displayed. Mutable reference types can be half-updated.

## Recipe 2.4 — Waiting for a Set of Tasks

**Problem:** You have multiple tasks running and need to wait for all of them.

**Solution:** `Task.WhenAll`

```csharp
// Wait for three independent delays
Task t1 = Task.Delay(TimeSpan.FromSeconds(1));
Task t2 = Task.Delay(TimeSpan.FromSeconds(2));
Task t3 = Task.Delay(TimeSpan.FromSeconds(1));
await Task.WhenAll(t1, t2, t3);  // completes in ~2s

// WhenAll returns results when all tasks have the same type
Task<int> r1 = Task.FromResult(3);
Task<int> r2 = Task.FromResult(5);
int[] results = await Task.WhenAll(r1, r2);  // [3, 5]

// Start multiple downloads concurrently, then collect all
async Task<string> DownloadAllAsync(HttpClient client, IEnumerable<string> urls)
{
    // ToArray() materializes the query — tasks start NOW
    Task<string>[] tasks = urls.Select(url => client.GetStringAsync(url))
                               .ToArray();
    string[] pages = await Task.WhenAll(tasks);
    return string.Concat(pages);
}
```

**Handling multiple exceptions:**
```csharp
async Task ObserveAllExceptionsAsync()
{
    var t1 = ThrowAsync<NotImplementedException>();
    var t2 = ThrowAsync<InvalidOperationException>();
    Task all = Task.WhenAll(t1, t2);
    try
    {
        await all;
    }
    catch
    {
        // all.Exception is AggregateException with BOTH exceptions
        AggregateException allErrors = all.Exception;
    }
}
```

> ⚠️ When you `await Task.WhenAll(...)`, only the **first** exception is rethrown. Inspect `task.Exception` directly to see all of them.

## Recipe 2.5 — Waiting for Any Task to Complete

**Problem:** You have multiple competing tasks and only care about the first to finish.

**Solution:** `Task.WhenAny`

```csharp
// Returns data from whichever URL responds first
async Task<int> FirstRespondingUrlAsync(HttpClient client, string urlA, string urlB)
{
    Task<byte[]> taskA = client.GetByteArrayAsync(urlA);
    Task<byte[]> taskB = client.GetByteArrayAsync(urlB);

    // completedTask is whichever finished first
    Task<byte[]> completedTask = await Task.WhenAny(taskA, taskB);

    // IMPORTANT: await the inner task to surface any exceptions
    byte[] data = await completedTask;
    return data.Length;
}
```

```
WhenAny timeline:
  taskA: ──────────────────[done]
  taskB: ──────────[done]
              ▲
              WhenAny completes here (with taskB as result)
              taskA continues running in background
```

> ⚠️ `WhenAny` never faults or cancels — the outer task always succeeds. Always await the inner (winning) task to propagate its exception.

> ⚠️ **Cancel the losing tasks** when possible. Abandoned tasks continue consuming resources (HTTP connections, timers, memory).

> ⚠️ **Don't use WhenAny to process tasks in completion order** — that's O(N²). Use Recipe 2.6 instead.

## Recipe 2.6 — Processing Tasks as They Complete

**Problem:** Process each task as soon as it finishes, not in the order they were started.

**Solution:** Wrap each task in a helper async method that awaits it and processes the result.

```csharp
async Task<int> DelayAndReturnAsync(int value)
{
    await Task.Delay(TimeSpan.FromSeconds(value));
    return value;
}

// ❌ BAD: processes in start order, not completion order
async Task ProcessInOrder()
{
    Task<int>[] tasks = { DelayAndReturnAsync(2), DelayAndReturnAsync(3), DelayAndReturnAsync(1) };
    foreach (var task in tasks)
        Console.WriteLine(await task);  // prints 2, 3, 1 — wrong order!
}

// ✅ GOOD: process each task as it completes (prints 1, 2, 3)
async Task ProcessByCompletion()
{
    Task<int>[] tasks = { DelayAndReturnAsync(2), DelayAndReturnAsync(3), DelayAndReturnAsync(1) };

    // Each of these helper tasks awaits one original task and processes its result
    Task[] processors = tasks.Select(async t =>
    {
        int result = await t;
        Console.WriteLine(result);
    }).ToArray();

    await Task.WhenAll(processors);
}
```

## Recipe 2.7 — Avoiding Context for Continuations

**Problem:** Your async method resumes on the UI context, causing performance problems.

**Solution:** `ConfigureAwait(false)`

```csharp
// Resumes on original context (UI thread if called from UI)
async Task ResumeOnContextAsync()
{
    await Task.Delay(1000);
    // This runs on UI thread — fine for UI code, wasteful for library code
}

// Resumes on any thread pool thread — better for library code
async Task ResumeWithoutContextAsync()
{
    await Task.Delay(1000).ConfigureAwait(false);
    // This runs on a thread pool thread
}
```

```
Context capture:
  UI Thread → calls async method → awaits → [operation completes]
      ↓                                            │
  Captured SynchronizationContext (UI)  ←──────────┘
      ↓
  Method resumes on UI thread (ConfigureAwait(true), the default)
  OR
  Method resumes on any thread pool thread (ConfigureAwait(false))
```

> 💡 **Rule:** Use `ConfigureAwait(false)` in all **library/infrastructure** code. Only resume on context in **UI/presentation** code that actually needs to touch UI elements.

> ⚠️ A thousand+ continuations per second on the UI thread will degrade performance. `ConfigureAwait(false)` is a simple fix.

## Recipe 2.8 — Handling Exceptions from async Task Methods

**Problem:** You need to handle exceptions from async methods correctly.

**Solution:** Use `try`/`catch` around `await` — it works exactly as expected.

```csharp
async Task TrySomethingAsync()
{
    try
    {
        await PossibleExceptionAsync();
    }
    catch (NotSupportedException ex)
    {
        LogException(ex);
        throw;  // re-throw with original stack trace preserved
    }
}
```

```
Exception flow:
  async method throws ──→ exception placed on returned Task
                                          │
                                          ▼
                           Task is awaited elsewhere
                                          │
                                          ▼
                           Exception re-thrown at the await site
                           (original stack trace preserved ✓)
```

> ⚠️ **Don't call `.Wait()` or `.Result` to get the exception.** They wrap it in `AggregateException`. Use `await` instead.

## Recipe 2.9 — Handling Exceptions from async void Methods

**Problem:** You have an `async void` method that throws — how do you catch it?

**Solution:** You mostly can't. **Avoid `async void`** whenever possible.

```csharp
// ❌ BAD: exception propagates to SynchronizationContext, hard to catch
async void MyVoidMethod()
{
    await Task.Delay(100);
    throw new InvalidOperationException("Lost forever");
}

// ✅ GOOD: return Task so the caller can await it
async Task MyTaskMethod()
{
    await Task.Delay(100);
    throw new InvalidOperationException("Caller can catch this");
}

// ✅ If you MUST use async void (event handlers only):
async void Button_Click(object sender, EventArgs e)
{
    try
    {
        await DoWorkAsync();  // all logic in a Task-returning method
    }
    catch (Exception ex)
    {
        MessageBox.Show(ex.Message);
    }
}
```

```
Where async void exceptions end up:
  async void method throws
         │
         ▼
  SynchronizationContext.Post (the current context's exception handler)
         │
         ▼
  WPF:    Application.DispatcherUnhandledException
  UWP:    Application.UnhandledException
  ASP.NET UseExceptionHandler middleware
  Console: app crashes
```

> 💡 **Only legitimate use of `async void`:** Event handlers (because `EventHandler` returns `void`). For all other cases, return `Task`.

## Recipe 2.10 & 2.11 — ValueTask

**Problem:** You need to create/consume `ValueTask<T>` (for performance-critical code or interface implementation).

**Solution for creating:**
```csharp
// Simplest: just use async/await as normal
public async ValueTask<int> MethodAsync()
{
    await Task.Delay(100);
    return 13;
}

// Optimized: avoid allocation when result is already available
public ValueTask<int> MethodAsync()
{
    if (CanBehaveSynchronously)
        return new ValueTask<int>(13);       // zero allocation
    return new ValueTask<int>(SlowMethodAsync());  // falls back to Task
}
```

**Solution for consuming:**
```csharp
// ✅ Simplest — just await it once
int value = await MethodAsync();

// ✅ Also fine — start it, do other work, then await
ValueTask<int> vt = MethodAsync();
// ... other work ...
int value = await vt;   // await ONLY ONCE

// ✅ If you need to do more complex things, convert to Task first
Task<int> task = MethodAsync().AsTask();
int a = await task;
int b = await task;  // safe — Task can be awaited multiple times
```

```
ValueTask rules:
  ┌─────────────────────────────────────────┐
  │ 1. Await it only ONCE                   │
  │ 2. Don't await AND call AsTask          │
  │ 3. Don't access .Result before complete │
  │ 4. When in doubt → convert to Task      │
  └─────────────────────────────────────────┘
```

> ⚠️ `ValueTask<T>` has strict rules. Unlike `Task<T>`, you cannot await it multiple times, store it and await later multiple times, or call both `await` and `AsTask()` on the same instance.

> 💡 **When to use `ValueTask<T>` vs `Task<T>`:** Use `Task<T>` by default. Only switch to `ValueTask<T>` after profiling shows a measurable benefit, or when implementing interfaces that require it (like `IAsyncDisposable.DisposeAsync()`).

## The Deadlock Pattern to Avoid

```csharp
// ❌ DEADLOCK on UI thread or ASP.NET Classic:
async Task WaitAsync()
{
    await Task.Delay(1000);  // captures UI context
}

void Deadlock()
{
    Task task = WaitAsync();
    task.Wait();  // blocks UI thread, which WaitAsync needs to resume → STUCK!
}
```

```
Deadlock sequence:
  UI Thread: calls WaitAsync() → await captures UI context
  UI Thread: calls task.Wait() → blocks
  [1 second passes]
  Continuation: tries to resume on UI context
  UI context: BLOCKED by task.Wait()
  → Deadlock!
```

**Two fixes:**
1. Use `ConfigureAwait(false)` inside `WaitAsync`
2. Make `Deadlock()` into an `async` method and `await` instead of `.Wait()`
